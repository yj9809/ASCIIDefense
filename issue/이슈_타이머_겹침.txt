[Enemy 겹침 이슈 정리 - 타이머 불일치]

1) 현재 증상
- Enemy끼리 겹쳐서 이동하는 프레임이 발생함.
- "겹칠 것 같으면 한 프레임 대기"가 의도대로 동작하지 않음.

2) 핵심 원인
- MoveEnemies()에서 승자 선정 후 willVacate(비움 예약)를 먼저 기록함.
- 하지만 실제 이동은 Enemy::Tick()의 개별 타이머(timer > 0.5f) 조건을 통과해야만 실행됨.
- 즉 "예약상 비우는 칸"과 "실제로 비워지는 칸"이 달라지는 프레임이 생김.
- 그 프레임에서 뒤 Enemy가 진입하면서 겹침이 발생함.

3) 왜 현재 구조에서 어긋나는가
- 예약 판단 시점: GameLevel::MoveEnemies()
- 실제 이동 시점: Enemy::Tick() 내부 개별 타이머 충족 시
- 두 시점/조건이 분리되어 있어, 예약 정확도가 깨짐.

4) 해결 방향 (권장)
A. 후보 제한
- toGroups(목적지 경쟁 후보)에는 "이번 프레임 실제 이동 가능한 Enemy"만 넣는다.
- 조건 예시: HasNext() && CanMoveNow()

B. 점유 검사는 전체 기준 유지
- fromMap(현재 점유 맵)은 모든 Enemy를 기준으로 만든다.
- 목적지(to)가 점유되어 있으면, 점유자가 이번 프레임 확실히 비우는 경우에만 진입 허용.

C. API 분리
- Enemy/Agent에 아래 성격의 조회 함수 추가:
  - HasNext(): 다음 경로가 남았는지
  - CanMoveNow(): 현재 타이머로 이번 프레임 이동 가능한지
- MoveEnemies()는 조회 결과를 기반으로 승자/차단만 결정.
- 실제 Move()는 기존 Tick 흐름 유지 가능.

5) 대안
- 개별 Enemy 타이머를 제거하고, GameLevel 단일 이동 주기로 통제하면 로직 단순화 가능.
- 단, 현재 요구(개별 속도 조절)와 다를 수 있어 우선은 후보 제한 방식이 적합.

6) 결론
- 문제의 본질은 "willVacate 예약"과 "실제 이동 가능 시점" 불일치.
- 후보를 CanMoveNow 기반으로 제한하면 겹침 현상이 크게 줄고, 규칙 일관성이 생긴다.
